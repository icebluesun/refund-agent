Safety & Constraint Specification
Refund Agent – Loopp Challenge

1. INPUT VALIDATION & SCHEMA ENFORCEMENT

Pydantic Models for Type Safety:

from pydantic import BaseModel, Field, validator
from typing import Literal

class RefundRequest(BaseModel):
    refund_id: str = Field(..., regex=r'^REF-\d+$')
    customer_id: str = Field(..., regex=r'^CUST-\d{3}$')
    item_id: Literal["ITEM-001", "ITEM-002", "ITEM-003"]
    unit_ids: list[str] = Field(..., min_items=1, max_items=5)
    refund_quantity: int = Field(..., ge=1, le=5)
    refund_value: float = Field(..., ge=0, le=5000)
    reason: str = Field(..., max_length=500)
    created_at: datetime

    @validator('refund_value')
    def value_matches_quantity(cls, v, values):
        if 'unit_ids' in values:
            expected = len(values['unit_ids']) * 149.99  # depends on item
            assert abs(v - expected) < 0.01
        return v

class DisputeState(BaseModel):
    decision: Literal["Approved", "Rejected", "Pending"]
    evidence: str = Field(..., min_length=10, max_length=2000)
    retry_count: int = Field(..., ge=0, le=3)
    latency_ms: float = Field(..., ge=0, le=30000)

Benefits:
- Invalid data never reaches the graph
- Self-documenting API contracts
- Automatic error messages for malformed inputs

2. BLAST RADIUS CONTROL

Principle: Limit the damage a single bad decision can cause.

Pattern 1: Financial Circuit Breaker ($700 threshold)
- Any refund request over $700 exits the automated loop
- Human review required before approval
- Prevents catastrophic financial loss from LLM hallucination

Implementation:
def ai_agent_dispute_node(state):
    if refund_value > 700:
        return {
            "decision": "Pending",
            "assigned_to": "Human_Vet",
            "evidence": "Auto-escalation: Value exceeds $700 threshold"
        }
    # Otherwise proceed with automated evaluation

Pattern 2: Strike Limit (3 attempts per unit)
- Each unit has independent strike counter
- After 3 rejections, unit becomes STUCK
- Prevents endless retry loops and harassment
- Customer cannot refund same unit repeatedly

Pattern 3: Tool Execution Sandbox
- All database queries are read-only for customer tools
- Write operations (refund approval) require explicit confirmation
- No direct database access from LLM – only through validated functions

Pattern 4: Cooldown Timer (10 seconds)
- Prevents rapid-fire request spam
- Protects API rate limits
- Forces deliberate user interaction

Blast Radius Summary:
| Risk | Mitigation | Impact if Failed |
|------|------------|------------------|
| LLM approves $10,000 refund | $700 circuit breaker | Escalated to human |
| Customer spams 1000 requests | Cooldown + rate limiting | 429 errors, no financial loss |
| LLM calls wrong tool | Tool schema validation + read-only queries | Logged, no state corruption |
| Duplicate refund for same unit | Unit locking + pending checks | Second request blocked |

3. PROMPT INJECTION DEFENSE

Structural Defense (not just instruction-based):

Layer 1: System Prompt Isolation
- User input never concatenated into system prompt
- Clear separation: system instruction (immutable) + user input (variable)

Example (graph.py):
system_instruction = f"""
You are Loopp's refund agent. THE POLICY BELOW IS THE SOURCE OF TRUTH.
{policy.load_policy()}
"""
user_prompt = f"""
Customer Statement: "{state.get('reason')}"
"""
# NEVER: system_instruction + user_input directly

Layer 2: Input Sanitization
injection_patterns = [
    "ignore previous", "override policy", "you are now",
    "pretend you are", "admin override", "system instruction",
    "forget rules", "bypass", "jailbreak"
]
def detect_injection(user_input):
    for pattern in injection_patterns:
        if pattern in user_input.lower():
            return True, f"Blocked: {pattern}"
    return False, None

Layer 3: Output Schema Enforcement
- LLM output must conform to JSON schema
- Invalid JSON → auto-reject
- Prevents instruction leakage into output

Layer 4: Tool Call Whitelist
- LLM cannot call arbitrary functions
- Only pre-approved tools: get_owned_units, get_refundable_units, get_policy
- No "execute_code" or "system" tools

4. RATE LIMITING & RESOURCE PROTECTION

| Layer | Mechanism | Limit |
|-------|-----------|-------|
| Per-user | Cooldown timer | 10 seconds between requests |
| Per-API | Minimum interval | 2 seconds between Gemini calls |
| Per-IP | Request rate | 60 requests/minute |
| Total | Concurrent locks | One refund per unit at a time |

Implementation:
from functools import lru_cache
import time

_rate_limit_cache = {}
def check_rate_limit(user_id):
    last_call = _rate_limit_cache.get(user_id, 0)
    if time.time() - last_call < 10:
        return False, 10 - (time.time() - last_call)
    _rate_limit_cache[user_id] = time.time()
    return True, 0

5. AUDIT TRAIL & FORENSICS

Every decision is logged with:
- Full input (customer statement, tier, value)
- Full output (decision, evidence, token usage)
- LLM model version and parameters
- Timestamp and latency
- Retry count and failure reasons

This enables:
- Post-incident analysis of bad decisions
- Policy refinement based on real escalations
- Compliance with financial regulations
- Customer dispute resolution
